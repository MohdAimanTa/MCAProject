01
Background
Context, problem statement & objectives

Background of the System

  • Physical security systems are a critical component of building access control in both residential and commercial environments.
  • Traditional lock-and-key mechanisms are increasingly being replaced by electronic and microcontroller-based solutions offering greater flexibility, auditability, and user management.
  • Keypad-based door lock systems are a cost-effective implementation commonly found in offices, laboratories, server rooms, and residential properties.
  • This project is developed within the domain of embedded systems and microprocessor architecture, leveraging the ARM Cortex-M4 based NUCLEO-F401RE running Mbed OS.
  • It draws on established techniques: PWM-controlled servo actuation, I2C display communication, and matrix keypad scanning.

Problem Statement

  • Conventional door locks rely on physical keys — which can be lost, duplicated, or forgotten — posing both security and usability risks.
  • There is a need for a low-cost, programmable electronic access control system that authenticates users via PIN, provides real-time visual feedback, and controls a physical locking mechanism.
  • The challenge lies in correctly integrating multiple peripherals (keypad, OLED, servo, LEDs, buzzer) through a single microcontroller while ensuring reliable debouncing, precise servo timing, and a clear user interface.
  • All of this must function within the constraints of an embedded real-time environment.

Project Objectives

What the system is intended to achieve
  1. Implement a PIN-based authentication system using a 4×4 matrix keypad that accepts, masks, and verifies user-entered passwords up to 8 characters long.
  2. Control an SG90 servo motor to physically lock and unlock a door mechanism with consistent and equal travel time in both directions.
  3. Display real-time system status, prompts, and feedback on an SSD1306 OLED display (128×32) via I2C communication.
  4. Provide audio-visual feedback using LEDs and a buzzer to indicate access granted, access denied, and alarm states.
  5. Allow authorised users to change the system password and provide a master override mode for administrative access without prior credentials.

Hardware Design
Architecture, components & circuit diagrams

System Block Diagram

  • NUCLEO-F401RE — central processing unit; all peripherals connect to this board.
  • 4×4 Matrix Keypad — user input via GPIO; rows as digital outputs (PC_0–PC_3), columns as pull-up digital inputs (PB_0–PB_3).
  • SSD1306 OLED (128×32) — connects via I2C (SDA: PB_9, SCL: PB_8) at address 0x3C.
  • SG90 Servo Motor — driven by PWM on PA_0 at 50 Hz; rotates between 0° (locked) and 90° (unlocked).
  • Green LED (D2) — GPIO digital output; indicates access granted.
  • Red LED (D3) — GPIO digital output; indicates access denied or alarm state.
  • Buzzer (D4) — GPIO digital output; provides audio feedback for all system states.

Hardware Architecture

  • Keypad scanning — row-drive / column-read: each row driven low in turn while columns monitored for a low signal indicating a key press.
  • OLED display — communicates over I2C at 0x3C via SSD1306 library, rendering text prompts and status across 4 lines at 3.3 V logic.
  • Servo motor — 50 Hz PWM (period = 20 ms); ~1.0 ms pulse (0°, locked), ~1.5 ms pulse (90°, unlocked).
  • Servo timing fix — snapped to origin for 300 ms before driving to target for 1000 ms; PWM then cut to eliminate idle current draw.
  • LEDs & buzzer — GPIO DigitalOut pins toggled with timed delays to produce distinct feedback patterns.

Circuit / Wiring Diagram

  • Keypad rows — PC_0–PC_3 as DigitalOut; driven low one at a time during scanning.
  • Keypad columns — PB_0–PB_3 as DigitalIn with internal pull-up; low reading indicates key press.
  • SSD1306 OLED — SDA on PB_9, SCL on PB_8; standard I2C at 3.3 V logic.
  • SG90 Servo — signal wire to PA_0 (PWM); VCC at 5 V; GND shared with board.
  • Green LED (D2) — in series with 220 Ω current-limiting resistor to GND.
  • Red LED (D3) — in series with 220 Ω current-limiting resistor to GND.
  • Buzzer (D4) — active buzzer connected directly between D4 and GND rail.
  • GND — all components share the board’s common GND rail.

Firmware / Software Development
Architecture, flowchart & code modules

Firmware Architecture

  • Language / framework — C/C++ using Mbed OS 6, developed in Keil Studio Cloud.
  • Haikal.c — main door lock logic: keypad scanning, OLED display, servo control, and password management.
  • Fasya.cpp — audio-visual feedback module: green LED, red LED, and buzzer signal patterns.
  • Servo functions — servoInit(), unlockDoor(), lockDoor().
  • Display — showMessage() renders up to 4 lines of text on the OLED.
  • Keypad input — getKey(), waitForKey(), enterPassword().
  • Password management — setNewPassword() with optional old-password verification.
  • Feedback signals — successSignal(), wrongSignal(), alarmSignal() in Fasya.cpp.
  • main() — infinite loop managing the LOCKED / UNLOCKED state machine.

Program Flowchart

  • Startup — servo initialised (PWM off); OLED shows "System Ready" for 2 seconds.
  • Password entry — keypad scanned; input masked as asterisks on OLED; '#' to confirm, '*' to backspace.
  • Special keys — 'A' triggers password change (requires old PIN); 'D' triggers master override (skips old PIN).
  • PIN verified — correct PIN → "Access Granted", successSignal(), unlockDoor().
  • Unlock sequence — servo snaps to 0° for 300 ms, drives to 90° for 1000 ms, then PWM cut.
  • Unlocked state — system polls for 'B' key; servo idle (no current draw).
  • Lock on 'B' — lockDoor() (90°→0°); OLED "Door Locked"; returns to LOCKED state.
  • Wrong PIN — OLED "Access Denied" for 2 seconds; wrongSignal() fires; loop repeats.

Major Software Modules

Keypad input / data acquisition
  • getKey() implements active-low row scanning — iterates through all four rows, driving each low while reading all four column pins.
  • Matching key character returned from keyMap[4][4] lookup table.
  • 20 ms software debounce delay applied after key release to prevent multiple detections.
  • waitForKey() blocks until a valid key is detected.
  • enterPassword() builds a character buffer with backspace support and renders masked asterisk display on OLED.
Signal processing / computation
  • Password verification uses strcmp() to compare entered string against storedPassword[9].
  • angleToPulse() converts a servo angle (0–180°) to PWM pulse width: pulsewidth = 0.001 + (angle / 180.0) × 0.001
  • Maps 0° → ~1.0 ms and 90° → ~1.5 ms, within the standard 1–2 ms servo control range.
Control logic
  • Two-state FSM: LOCKED and UNLOCKED.
  • LOCKED — continuously prompts for PIN; correct → UNLOCKED; incorrect → denial message and repeat.
  • UNLOCKED — polls for 'B'; on press → lockDoor() → returns to LOCKED.
  • Special keys 'A' and 'D' invoke password change sub-routine (with or without old PIN verification).
Display / audio-visual feedback
  • showMessage() clears OLED and renders up to 4 lines using SSD1306 printString(), then calls display().
  • successSignal() — green LED on + 200 ms beep for 2 seconds (access granted).
  • wrongSignal() — red LED and buzzer flash 3× at 300 ms intervals (wrong PIN).
  • alarmSignal() — red LED and buzzer flash 10× at 500 ms intervals (alarm / repeated failure).
  • shortBeep(ms) helper abstracts buzzer on/off timing into a reusable function.

Testing, results, achievements & future work

Project Demo Video

System Testing Procedures

  • Keypad scanning — all 16 keys verified by printing detected characters to OLED; no ghosting or debounce issues observed.
  • Password masking — entered known PINs and confirmed OLED showed correct number of asterisks.
  • Correct PIN (default "1234") — verified servo unlocks and OLED shows "Access Granted".
  • Incorrect PIN — verified denial screen appears and servo does not move.
  • Servo timing symmetry — lock and unlock travel time measured visually; both directions took approximately 1 second.
  • Password change ('A') — old password verified correctly; new password accepted after matching confirmation; old PIN rejected on next attempt.
  • Master override ('D') — new password set without requiring old PIN.
  • LED & buzzer patterns — success, wrong, and alarm signals triggered and confirmed for correct pattern durations.

Project Achievements

✓
Fully functional PIN-based authentication system — accepts, masks, verifies, and rejects passwords up to 8 characters using a 4×4 matrix keypad with reliable debouncing.
✓
Servo timing symmetry problem identified and solved — two-step snap-then-drive approach ensures equal physical travel time in both lock and unlock directions.
✓
Complete OLED user interface implemented — real-time prompts, masked PIN entry, and all status messages rendered clearly on the 128×32 SSD1306 display.
✓
Multi-level access control achieved — standard users change password via 'A' (requires old PIN); master override 'D' allows admin reset without prior credentials.
✓
Distinct audio-visual feedback signals implemented — three clearly distinguishable patterns (success, wrong PIN, alarm) using green/red LEDs and buzzer.

All five project objectives were successfully met. The Security Door Lock System performs reliably as an integrated embedded solution — the NUCLEO-F401RE correctly orchestrates all peripherals through a clean state machine, demonstrating practical application of GPIO control, I2C communication, PWM actuation, and real-time user interaction within the Mbed OS framework.

Limitations & Constraints

  1. Password stored in RAM as plain-text (storedPassword[]) — lost on power cycle, reverting to default "1234". No persistent non-volatile storage implemented.
  2. No lockout mechanism after repeated failed PIN attempts — brute-force attack is possible without a timed lockout.
  3. Blocking wait_ms() / wait_us() calls halt the processor during servo movement and feedback delays, preventing concurrent tasks.
  4. Keypad scanning relies on polling with a 20 ms debounce delay rather than interrupt-driven input — may occasionally miss rapid key presses.
  5. SG90 servo provides limited torque — suitable only for lightweight door latch mechanisms; heavier locks require a higher-torque actuator.

Future Improvements

  1. Implement non-volatile password storage using STM32 internal flash or an external I2C EEPROM (e.g. AT24C32) so the password persists across power cycles.
  2. Add failed-attempt lockout — after 3 consecutive wrong PINs, disable input for a configurable timeout and trigger the alarm signal.
  3. Replace blocking wait_ms() calls with Mbed OS RTOS threads or event queues to allow concurrent keypad monitoring during servo and feedback sequences.
  4. Implement an activity log via UART serial output or SD card, recording timestamps and access events (granted / denied) for auditing.
  5. Add a proximity sensor (PIR or RFID reader) as a secondary authentication factor, or as an auto-lock trigger when the door area is vacated.
